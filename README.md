## Общая архитектура

Проект представляет собой **набор микросервисов**, обеспечивающих полную функциональность платформы **Explore With Me**. Архитектура построена на принципах:

- **Масштабируемость** – каждый микросервис можно развернуть независимо, горизонтально масштабируя только нужные компоненты.
- **Устойчивость** – сбой одного сервиса не приводит к падению всего приложения (изоляция ошибок).
- **Гибкость** – изменения в одном сервисе не затрагивают остальные, упрощая разработку и тестирование.

**Инфраструктурные компоненты:**
- **API Gateway (Gateway Server)** – единая точка входа для всех клиентов. Маршрутизирует запросы к соответствующим микросервисам.
- **Config Server** – централизованное хранилище конфигураций для всех сервисов (поддерживает профили).
- **Discovery Server (Eureka)** – сервис обнаружения, позволяющий микросервисам динамически регистрироваться и находить друг друга.

**Взаимодействие между микросервисами** реализовано через **OpenFeign** с балансировкой нагрузки (Spring Cloud LoadBalancer) и регистрацией в Eureka.

## Микросервисы

| Сервис | Описание | Основные эндпоинты (внешние) |
|--------|----------|------------------------------|
| **event-service** | Управление событиями (создание, обновление, поиск, публикация) | `/admin/events`, `/events`, `/users/{userId}/events` |
| **user-service** | Администрирование пользователей | `/admin/users` |
| **category-service** | Управление категориями событий | `/admin/categories`, `/categories` |
| **request-service** | Управление заявками на участие в событиях | `/users/{userId}/requests` |
| **rating-service** | Управление рейтингами (лайки/дизлайки) | `/private/events/**`, `/public/events/**` |
| **compilation-service** | Управление подборками событий | `/admin/compilations`, `/compilations` |

## Инфраструктура

### API Gateway (порт 8080)
- Принимает все внешние запросы и перенаправляет их в соответствующие микросервисы.
- Маршруты настраиваются динамически через `config-server` и кэшируются локально.
- Использует Eureka для определения актуальных адресов сервисов (`lb://`).

### Config Server
- Хранит конфигурационные файлы всех микросервисов в `resources/config/`.
- Поддерживает профили (`application.yml`, `application-H2.yml` и т.д.).
- Позволяет обновлять настройки без перезапуска сервисов (при использовании Spring Cloud Bus, но в текущей реализации обновление требует перезапуска).

### Discovery Server (Eureka, порт 8761)
- Все микросервисы регистрируются в Eureka при старте.
- Gateway использует Eureka для динамического поиска сервисов по имени (например, `lb://event-service`).
- Обеспечивает отказоустойчивость – при падении одного экземпляра запросы автоматически перенаправляются на другой (если запущено несколько экземпляров).

## Взаимодействие микросервисов

Для обмена данными между микросервисами используются **Feign-клиенты** с интерфейсами, аннотированными `@FeignClient(name = "service-name")`.

| Вызывающий сервис | Вызываемый сервис | Цель вызова |
|-------------------|-------------------|-------------|
| `event-service` | `user-service` | Получение информации о пользователе (имя, id) для заполнения DTO |
| `event-service` | `category-service` | Получение категории по id |
| `event-service` | `request-service` | Получение количества подтверждённых заявок на событие |
| `event-service` | `rating-service` | Получение статистики рейтинга события (like/dislike) |
| `request-service` | `user-service` | Проверка существования пользователя |
| `request-service` | `event-service` | Проверка существования события, получение его состояния и лимита участников |
| `rating-service` | `user-service` | Проверка существования пользователя |
| `rating-service` | `event-service` | Проверка существования события, получение даты проведения |
| `rating-service` | `request-service` | Проверка, участвовал ли пользователь в событии (статус CONFIRMED) |
| `compilation-service` | `event-service` | Получение событий по списку id для формирования подборки |

Все внутренние запросы проходят через **внутренние API** (`/internal/**`), чтобы отделить их от внешнего интерфейса и избежать случайного доступа извне.

## Внутреннее API

Эндпоинты, доступные только для других микросервисов (не для конечных пользователей).

### User Service (`/internal/users`)
| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/internal/users/{userId}` | Получить информацию о пользователе (id, имя) |

### Category Service (`/internal/categories`)
| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/internal/categories/{catId}` | Получить категорию по id |

### Request Service (`/internal/requests`)
| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/internal/requests/count/{eventId}` | Количество подтверждённых заявок на событие |
| GET | `/internal/requests/exists/{eventId}/{userId}` | Проверить, участвовал ли пользователь в событии (статус CONFIRMED) |
| GET | `/internal/requests/event/{eventId}` | Получить все заявки на событие (без проверки прав) |
| PATCH | `/internal/requests/event/{eventId}/status` | Обновить статусы заявок (без проверки прав, предполагается, что проверка выполнена вызывающим сервисом) |

### Rating Service (`/internal/ratings`)
| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/internal/ratings/{eventId}/stats` | Получить статистику рейтинга события (like/dislike) |

### Event Service (`/internal/events`)
| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/internal/events/{eventId}` | Получить событие (без проверки прав) |
| GET | `/internal/events/by-ids` | Получить список событий по списку id |
| GET | `/internal/events/{eventId}/check-published` | Проверить, опубликовано ли событие |
